# CBID Quick Reference Guide
## Constraint-Based Infrastructure Discovery

### Core Principles
1. **Constraints are data sources** - not just problems to solve
2. **Timing patterns are the universal language** of system architecture  
3. **Velocity mismatches reveal system boundaries** - different domains operate at different speeds
4. **Multi-domain observation enables triangulation** - different perspectives reveal different patterns
5. **Systematic documentation transforms frustration into curiosity**

### The 7-Step CBID Process
1. **Constraint Identification** - Document timing discrepancies, velocity mismatches
2. **Multi-Domain Observation** - Monitor across all accessible domains
3. **Pattern Recognition** - Identify threshold, interval, split, velocity, coordination patterns
4. **Infrastructure Mapping** - Use patterns to reveal system architecture
5. **Hypothesis Development** - Create testable predictions from patterns
6. **Verification & Refinement** - Test hypotheses, learn from results
7. **Documentation & Application** - Create constraint logs, case studies, methodology improvements

### Constraint Types
- **Physical**: Human/transportation scale (minutes-hours)
- **Digital**: Computational scale (seconds-minutes)  
- **Infrastructure**: Cache/propagation scale (seconds-minutes)
- **Cognitive**: Psychological thresholds, expectation patterns
- **Temporal**: Historical recurrence, seasonal patterns

### Key Pattern Categories
1. **Threshold Patterns**: Psychological (60-min), system limits
2. **Interval Patterns**: Regular, irregular, coordination cadences
3. **Split Patterns**: Cache boundaries, coordination delays, domain velocity differences
4. **Velocity Patterns**: Ratios (8:1), acceleration/deceleration
5. **Emergent Patterns**: Specialization, historical echoes, archetypes

### Essential Questions
- What timing patterns does this constraint create?
- Which domains show the constraint, and which don't?
- What velocity mismatches are observable?
- What system boundaries do split patterns reveal?
- What testable hypotheses can we develop from patterns?
- How can we verify our understanding?

### Common Insights
- **CDN layers**: Timing splits reveal cache TTLs (300s vs 600s)
- **Coordination protocols**: Sequential updates show hidden dependencies
- **Specialization emergence**: Constraints drive role development
- **Historical patterns**: Constraints recur in similar contexts
- **Metabolism timing**: Systems have characteristic operation speeds

### Value Proposition
- **Transforms**: Frustration → Curiosity, Problems → Data, Black boxes → Understandable systems
- **Reveals**: Hidden architecture, coordination patterns, system boundaries
- **Enables**: Prediction, optimization, accelerated understanding
- **Applies**: System debugging, team coordination, infrastructure discovery, historical analysis

### Getting Started
1. Choose a current constraint
2. Document across multiple domains
3. Look for timing patterns
4. Develop testable hypotheses
5. Create a constraint log
6. Share your learning

### Remember
Every constraint is speaking to you in the language of timing patterns. Your job is to listen, learn, and discover.
